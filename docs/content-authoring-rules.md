# 자격증 컨텐츠 제작 규칙 (프롬프트용)

이 문서는 자격증 문항·모의구성·UI 카피를 만들 때 **그대로 복사해 AI 또는 작성자에게 지시**하는 용도다.

---

## 1. 저작·법적 경계

- **실제 기출·공개 모의고사·유료 교재의 문항을 불법적으로 동일 복제하지 않는다.**
- **출처 문장·선지·도표를 그대로 가져오는 행위는 금지.**
- **출제 의도·범위는 참고하되, 문장·수치·선지 구성은 반드시 창작하거나 충분히 변형한다.**
- “비슷한 유형·난이도의 **원문과 다른** 새 문항” 작성은 허용한다.

---

## 2. 시험 구성·퀄리티 기준

- 대상이 **실제 필기/실기·공식 모의고사**가 있다면, **문항 수·시험 시간·과목(영역) 배분·배점 구조**를 먼저 조사해 **동일한 형식의 풀세트 모의**를 목표로 한다.
- 앱 내 **랜덤 N문항**만 뽑더라도, **전체 풀**은 “실전 모의 한 세트와 같은 밀도·범위”를 갖춘다.
- 선지는 **4지선다** 등 실제 시험 형식을 따른다.
- 해설에는 **오답을 고르기 쉬운 이유** 또는 **핵심 개념 한 줄**을 넣는다.

---

## 3. 난이도 3단계 (필수)

모든 문항에 아래 중 하나를 부여한다. JSON 필드 `difficulty`에 사용한다.

| 값 | 의미 |
|----|------|
| `easy` | 기본 용어·개념·직관적 응용. 대다수 수험생이 빠르게 풀 수 있는 난이도. |
| `medium` | 개념을 연결하거나 비교·적용해야 하는 난이도. |
| `hard` | 함정 선지·계산·세부 규격·오답률이 높을 수 있는 유형에 가깝게 설계. |

- **한 세트(예: 40문항 풀)** 안에서 난이도 비율을 맞춘다: **대략 easy : medium : hard = 3 : 4 : 3** (±2문항 범위에서 조정 가능).
- 난이도는 **문항 내용**에 근거해 일관되게 매긴다.

---

## 4. 데이터 필드 (JSON)

```json
{
  "id": 1,
  "question": "문항 본문",
  "options": ["A선지", "B선지", "C선지", "D선지"],
  "answer": 0,
  "explanation": "정답 근거·핵심 개념",
  "category": "영역-키 (영문 슬러그)",
  "difficulty": "easy"
}
```

- `answer`: 정답 선지의 **0부터 시작하는 인덱스**.
- `category`: 서비스에서 정한 과목/단원 키와 일치시킨다.

---

## 5. 제품 UI (참고)

- 랜딩/브랜드는 **특정 시험명만 단독 타이틀로 고정하지 않고**, **「자격증 도우미」** 등 포괄 명칭을 쓴다.
- **자격증 메뉴**에서 시험을 고른 뒤 해당 모의로 진입하는 구조를 기본으로 한다.

---

## 6. 복사용 프롬프트 (한 블록)

아래를 그대로 붙여 사용한다.

```
역할: 자격증 시험용 객관식 문항 작성자.

규칙:
1) 실제 기출·모의고사·교재 문항을 동일 복제하지 말 것. 문장·선지는 창작 또는 충분히 변형.
2) [시험명]의 공식·일반적으로 알려진 필기 시험 형식(문항 수, 시간, 과목 배분)에 맞춘 풀세트 퀄리티로 작성할 것.
3) 모든 문항에 difficulty를 부여: easy | medium | hard. 의미는 쉬운/중간/어려움·고오답 유형. 전체 비율은 easy:medium:hard ≈ 3:4:3.
4) 4지선다, answer는 0~3 정수 인덱스, explanation 필수, category는 [과목 키 목록] 중 하나.

산출물: JSON 배열, 스키마는 프로젝트 docs/content-authoring-rules.md 4절과 동일.
```

`[시험명]`, `[과목 키 목록]`만 채워 넣는다.

---

## 7. 정보처리기사 필기 (참고 수치 · 풀 설계 시)

- 통상 **객관식 필기**는 **과목별 문항 수·총 문항 수**가 고정되어 있다. 풀세트를 만들 때 **최신 공고·요강**의 수치를 반영한다.
- 본 저장소의 **10문항 랜덤 모의**는 위 풀에서 추출하는 방식이며, **풀 자체**가 모의고사 수준을 목표로 한다.
