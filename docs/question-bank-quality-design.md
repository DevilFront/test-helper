# 문항 뱅크 품질 설계 (전면)

> 목표: **실제 CBT·타사 모의고사에 가깝게** 느껴지도록, 출제 **규격·유형·검수**를 먼저 고정하고 데이터와 프로세스를 맞춘다.  
> 본 문서는 **코드 스키마 변경 전**에 합의할 **단일 기준서**다.

---

## 1. 목표와 비목표

### 1.1 목표

- 사용자가 **“짧고 가벼운 퀴즈”가 아니라 “시험에 가까운 모의”**라고 느끼게 한다.
- **과목·난이도·문항 유형** 분포가 공개 출제 경향과 **과도하게 어긋나지 않게** 관리한다.
- **오답 보기(방해선지)**가 그럴듯하고, 정답이 **근거 있게** 설명된다.
- 문항은 **버전·검수 상태**를 가지며, 문제가 있으면 **추적·교체** 가능하다.

### 1.2 비목표 (명시)

- **실제 기출문항의 문장 복제**는 하지 않는다 (권리·정책). **출제 경향·난이도·지문 길이**는 참고한다.
- “전문 출제자 1인이 모든 문항을 직접 쓴 것”과 **완전 동일**한 퀄을 **자동만으로** 보장하지 않는다. **사람 검수 또는 최소한의 규칙 검증**을 전제로 한다.

---

## 2. 품질의 정의 (사용자 체감)

| 차원 | 낮은 품질 (피해야 할 것) | 높은 품질 (지향) |
|------|-------------------------|------------------|
| **지문** | 한 문장 개념만, 맥락 없음 | 상황·조건·제한이 있어 **읽을 거리**가 있음 (필요 시 표/코드 블록) |
| **선지** | 정답만 맞고 나머지는 엉뚱 | **오답도 그럴듯** (개념 혼동·부분 진리·범위 틀림) |
| **난이도** | “hard” 라벨만 붙음 | **사고 깊이·계산·문맥**이 난이도와 일치 |
| **해설** | 정답만 한 줄 | **왜 정답인지 + 왜 다른 선지는 아닌지** (핵심만) |
| **일관성** | 시험명과 무관한 영어·지나친 줄임 | **용어·표기**를 시험/교재 관행에 맞춤 |

## 3. 문항 유형 분류 (Taxonomy)

실제 서비스에서는 **하나의 `question` 문자열**만으로는 부족하므로, 논리적으로 다음을 구분한다.

### 3.1 발문 형식 (`stemKind`)

| 코드 | 설명 | 사용자 체감 |
|------|------|-------------|
| `definition` | 용어·정의 직접 질문 | 짧고 빠름 |
| `which_best` | “가장 적절한 것은?” | 중간 |
| `scenario` | 상황 서술 후 판단 | **길고 실전에 가까움** |
| `code_or_sql` | 코드/SQL/의사코드 제시 | 실무·CBT 유사 |
| `table_diagram` | 표·간단 다이어그램 기반 | 읽기 부담 있음 |
| `not` | “옳지 **않은** 것은?” | 실수 유발, 검수 필요 |
| `combo` | 2단계 판단·조합 | 상위 난이도 |

**출제 비율**: 시험별로 목표 비율표를 둔다 (예: 정보처리 필기형 모의 → `scenario`+`which_best` 합이 전체의 X% 이상).

### 3.2 인지 수준 (`cognitiveLevel`) — 선택 메타

Bloom 단순화 예:

- `remember` — 암기·재인출  
- `understand` — 개념 설명·비교  
- `apply` — 절차·공식·코드 적용  
- `analyze` — 조건 비교·오류 찾기  

`difficulty`(easy/medium/hard)와 **독립**으로 두면, “쉬운 암기” vs “어려운 적용”을 구분해 풀 밸런스를 맞출 수 있다.

## 4. 데이터 모델 (목표 스키마 v2)

현재 `ExamQuestion`은 최소 필드만 있다. **하위 호환**을 위해:

- **v1**: 지금 구조 그대로 동작 (앱은 `question` 하나로 렌더).
- **v2**: 아래 필드가 있으면 **조합해** 지문을 구성한다.

### 4.1 필드 제안

```text
id
category
difficulty              # easy | medium | hard (기존)
cognitiveLevel?           # remember | understand | apply | analyze
stemKind?               # definition | scenario | ...

# 지문 (하나 이상 조합)
leadIn?                 # 상황/배경 (긴 문단 허용)
question                # 핵심 질문 (필수, 기존과 동일 역할)
contextBlock?           # 표/코드/다이어그램용 **원문 텍스트** (마크다운 제한적 또는 순수 텍스트)
contextType?            # none | plain | code | table

options[]
answer                  # 인덱스 (기존)
explanation             # 해설 (기존)

# 품질·운영 메타
sourceLevel?            # 기존
itemVersion?            # 정수, 문항 수정 시 증가
reviewStatus?           # draft | reviewed | published
reviewedAt?             # ISO 날짜 (선택)
tags?                   # ["정규화","이상현상"] 등 검색·중복 점검용
```

**렌더링 규칙 (앱)**:

1. `contextBlock`이 있으면 **질문 위에** 회색 박스 등으로 표시.
2. `leadIn`이 있으면 `question` 위에 표시.
3. 둘 다 없으면 기존처럼 `question`만 표시.

### 4.2 선지 규칙

- 4지선다 유지 (시험 형식 맞춤).
- 각 선지는 **동일한 문법 종결** (“~이다” / “~한 것은” 통일).
- **정답과 중복 의미**인 선지 금지.
- `not` 유형은 검수 시 **정답 인덱스 이중 확인** 필수.

## 5. 난이도 기준 (내부 루브릭)

라벨은 사람이 붙되, 기준을 문서화한다.

| 라벨 | 지문 | 사고 | 오답 |
|------|------|------|------|
| **easy** | 짧거나 단일 개념 | 기억·단순 대응 | 명확히 틀린 것 3개 |
| **medium** | 조건 1~2개 또는 짧은 시나리오 | 비교·간단 적용 | 1~2개는 헷갈릴 수 있음 |
| **hard** | 긴 시나리오·코드·예외 상황 | 다단계 판단·세부 구분 | 그럴듯한 오답 3개 |

**금지**: 지문은 쉬운데 라벨만 hard인 경우 → 검수에서 걸러낸다.

## 6. 커버리지 매트릭스

### 6.1 행: `category` (기존 JSON 키 유지)

### 6.2 열: `stemKind` 또는 `cognitiveLevel`

### 6.3 셀: 목표 문항 수 (또는 비율)

예: 정보처리기사 모의 풀 총 N문항일 때

- 과목별 최소 비율 (데이터베이스 X%, 네트워크 Y% …)
- `scenario` 최소 비율 Z%
- `hard` 전체 비율 상한/하한 (너무 어렵거나 쉬우면 조정)

**툴**: 스프레드시트 1장 + 빌드 시 스크립트로 **빈 셀 경고** (CI 또는 로컬).

## 7. 출제·검수 워크플로

```
[기획] 커버리지 표 확정
   ↓
[작성] 초안 (사람 또는 LLM 초안 + 템플릿)
   ↓
[자동 검사] 스키마·필드 누락·선지 길이 편차·금칙어(선택)
   ↓
[검수1] 사실·용어·난이도 라벨 적합성
   ↓
[검수2] (선택) 교차 검토
   ↓
[published] JSON 반영, itemVersion++
```

### 7.1 LLM의 위치

- **초안 생성·해설 확장·오답 후보 생성**에만 사용.
- **최종 게시 전** 반드시 사람이 `reviewStatus: reviewed`로 바꾸는 것을 원칙으로 한다.
- 자세한 API·보안은 `docs/openai-integration.md`와 동일 원칙.

## 8. 문항 단위 QA 체크리스트 (출시 전 필수)

각 문항에 대해 검수자가 확인:

- [ ] 지문만 읽고도 **질문 요구가 명확**한가?
- [ ] 정답이 **해설·교재 근거**와 모순되지 않는가?
- [ ] 오답 3개가 **무의미한 엉뚱함**이 아닌가?
- [ ] `not` 유형이면 **정답 인덱스**가 맞는가?
- [ ] 코드/표가 있으면 **복사 오류·줄바꿈**이 없는가?
- [ ] 난이도 라벨이 **지문·사고 깊이**와 맞는가?
- [ ] 용어가 **시험 공지 용어**와 크게 어긋나지 않는가? (개정 시 `tags`로 추적)

## 9. 마이그레이션 전략 (단계)

| 단계 | 내용 |
|------|------|
| **0** | 본 문서 합의, 샘플 문항 5~10개를 **v2 필드 포함**으로 작성해 UI·가독성 확인 |
| **1** | 타입 정의(`ExamQuestion` 확장) + 렌더러가 `leadIn`/`contextBlock` 지원 |
| **2** | 기존 문항은 v1 유지, **신규만 v2**부터 엄격 적용 |
| **3** | 우선순위 높은 과목부터 기존 문항 **리라이트** (짧은 문항 → 시나리오 보강 등) |
| **4** | 커버리지 스크립트(`npm run check-coverage`) + `reviewStatus` 게이트 (예: published만 랜덤 출제 옵션) — 스크립트는 §13 반영, 게이트는 미구현 |

## 10. 성공 지표 (출시 후)

- 세션당 **이탈률**·**재시도율** (모의 반복)
- 과목별 **정답률 분포** (한 과목만 극단적으로 쉽거나 어렵지 않은지)
- 사용자 피드백 “문항이 이상하다” **비율** (문항 id 추적)

---

## 11. 요약

- **짧은 지문**은 “틀린 것”이 아니라 **현재 풀의 기본 형태가 가볍게 잡혀 있기 때문**이다.
- 퀄리티를 올리려면 **유형·인지·지문 구조·검수**를 먼저 고정하고, 데이터와 UI를 **v2 스키마**에 맞추는 것이 정석이다.
- **LLM은 보조**, **규격과 사람 검수**가 본체다.

이 설계에 맞춰 다음 작업은 **① 타입·렌더러 확장 ② 샘플 v2 문항 시리즈 ③ 커버리지 검사 스크립트** 순으로 진행하는 것을 권장한다.

## 12. 구현 반영 현황 (코드·데이터)

- **타입**: `lib/types.ts` — `StemKind`, `CognitiveLevel`, `ContextType`, `ReviewStatus`, `leadIn`, `contextBlock`, `itemVersion`, `tags` 등.
- **메타·추론**: `lib/question-meta.ts` — 라벨, `inferStemKind` / `resolve*` (JSON 누락 시).
- **UI**: `components/question-body.tsx` — `leadIn` → `contextBlock` → `question` 순, 유형·인지 배지(선택).
- **연결**: 모의(`test-runner`), 결과 해설(`result-client`), 북마크 목록.
- **일괄 보강**: `npm run enrich-questions` → `stemKind`, `cognitiveLevel`, `itemVersion`, `reviewStatus` 자동 부여 (휴리스틱).
- **수동 고품질 샘플**: 정보처리 id 1·2·3(상황·해설 보강·코드 블록·태그), 산업안전 id 1(상황형 `leadIn`). 나머지는 휴리스틱 메타 + 추후 리라이트.
- **커버리지 점검**: `npm run check-coverage` — 시험별 과목·`stemKind`·난이도 집계, `category × stemKind` 표, 기대 과목 누락·상황형·부정형·코드형 하한 등 경고. CI에서 실패시키려면 `npm run check-coverage -- --fail-on-warn` (현재 데이터는 리라이트 전까지 경고가 남을 수 있음).

## 13. 커버리지 스크립트 (구현)

| 항목 | 내용 |
|------|------|
| 공통 | `scripts/coverage-shared.mjs` — 임계값·`analyzeExam`·`computeStemGaps` (check / gap-plan 동일 기준) |
| 점검 | `scripts/check-question-coverage.mjs` |
| npm | `npm run check-coverage` / `--fail-on-warn` |
| 기대 과목 | `info-processing`·`industrial-safety`별로 `lib/category-labels.ts` 키와 동일한 목록을 스크립트에 고정 |
| 출력 | 난이도·stemKind·인지·과목별 건수, 과목×stemKind 행렬, 경고·안내 |

§6의 “빈 셀 경고”에 대응: **전역**으로 상황형·부정형 최소치를 보고, 과목 단위로 `scenario`·`not`·`code_or_sql`가 모두 0인 경우 안내(ℹ)한다.

### 13.1 갭 대응 작업 계획 (후보 ID 선별)

- **스크립트**: `scripts/coverage-gap-plan.mjs` → `npm run coverage-gap-plan`
- **산출물**: 기본 `reports/coverage-gap-plan.json` (또는 `--out`, `--stdout`)
- **역할**: `check-coverage`와 같은 숫자 갭을 계산한 뒤, **어떤 `id`를 먼저 리라이트할지** 후보를 휴리스틱으로 나열한다.
- **후보 선정 로직**은 `scripts/coverage-candidates.mjs`에서 gap-plan과 공유한다.

### 13.2 부분 자동 수정 (기본 dry-run)

- **스크립트**: `scripts/apply-coverage-rewrites.mjs` → `npm run apply-coverage-rewrites`
- **기본**: 변경만 출력(dry-run), **파일에 쓰려면** `--apply`
- **옵션**: `--exam=info-processing` | `industrial-safety` | 생략 시 전체
- **하는 일**
  - **scenario**: `trySplitLeadIn()`으로 지문을 `leadIn` / `question`으로 나눌 수 있을 때만 적용. 실패 시 스킵(수동).
  - **code_or_sql**: `contextBlock`이 비어 있을 때만 SQL 스텁 + `contextType` + `stemKind` (반드시 문항에 맞게 교체·검수).
  - **not**: **자동 적용 안 함** — 정답·선지 뒤집힘 위험. 후보 id만 로그.
- **메타**: 수정된 문항에 `tags`에 `auto-coverage-rewrite` 추가, `itemVersion` 증가.

---

## 14. 자동 수정에 대해 (기대치 정리)

- **라벨만 맞추는 스크립트**(`stemKind`만 바꾸고 지문은 그대로)는 넣지 않는다.
- **§13.2**는 지문 구조를 분리하거나 SQL 스텁을 넣는 **제한적 기계적 수정**이며, **부정형·선지·해설의 의미 검수는 사람**이 담당한다.
- **가능한 자동화**: 감지·갭 수량·후보 ID·작업 힌트(§13.1), §13.2의 부분 적용, 스키마 검증, 선지 개수·인덱스 범위 검사 등.
- **반자동**: LLM으로 초안·해설 확장 후 `reviewStatus`·`itemVersion`으로 관리(설계 §7·`docs/openai-integration.md` 원칙).
